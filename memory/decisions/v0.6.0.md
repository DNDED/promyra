# Decisions — pi-pro v0.6.0

## Agent Swarm v1 release

### New `packages/swarm` (not a subagent module)
- **Why:** Swarm surface is too large for `packages/subagent` (orchestrator, 5 subagents, scratchpad, worktree-pool, verification, budget, types). Separate package enforces a clean boundary.
- **Captured:** `packages/swarm/src/{orchestrator,scratchpad,worktree-pool,verification-gate,budget,types}.ts`.

### File scratchpad, not message bus
- **Why:** Research-backed. For coding tasks, file-based beats in-memory message bus on durability, diffability, and restarts. Anthropic's Magentic-One and the OpenHands SDK both use file blackboards.
- **Decision:** All subagent outputs at `.pi-pro/swarm/<id>/<role>.md`. Cost accumulator at `cost.json`. State machine at `state.json`. Each subagent reads inputs from the scratchpad before running; writes its output after.
- **Captured:** spec §"Communication (file scratchpad)".

### 3 parallel stages max; otherwise sequential
- **Why:** Subagents with shared state (build → test → critique) must be sequential. Subagents with independent inputs (researcher, planner) can run in parallel. Deep parallelism (4+ stages) is hard to reason about and tends to lose to single-agent on SWE-bench.
- **Decision:** Topo: PLAN → RESEARCH+PLANNER (parallel) → BUILD → TEST → CRITIQUE → DECIDE → MERGE. The whole swarm is 3 LLM calls wide at peak.
- **Captured:** spec §"Swarm topology (orchestrator sequence)".

### Per-role worktrees, isolated writes
- **Why:** Worktree-per-subagent prevents race conditions when multiple subagents edit the same file. Read-only subagents (planner, researcher, critic) don't need worktrees.
- **Decision:** Builder + test-runner get git worktrees under `.pi-pro/worktrees/<id>/<role>/`. Planner/researcher/critic read cwd. At end, `pi swarm --merge <id>` merges builder's branch back to main.
- **Captured:** spec §"Worktree layout".

### Cost cap: soft warn at 50%, hard kill at 100%, $2 default
- **Why:** Research-backed. Magentic-One documents subagent loop stalls ~10-20% of the time. Without a hard cap, a runaway cascade can drain budget. Soft warn at 50% gives visibility before the kill.
- **Decision:** Per-swarm budget default $2.00; configurable via `--budget=<usd>`. Soft warn at 50% (log + TUI badge). Hard kill at 100% (current LLM call finishes, then halt with `paused: budget exceeded`). Resume with `--budget=<higher>`.
- **Captured:** spec §"Cost cap (per swarm)" + `packages/swarm/src/budget.ts`.

### Retry-with-feedback, default 2 retries
- **Why:** v0.4.0 already has `retryFeedback` in `apps/pi/src/commands/start.ts`. Reuse the pattern at swarm level. Critic-reviser (more expensive) is overkill for v0.6.0.
- **Decision:** Per-subagent retry policy. `builder` gets 2 retries; `test-runner` gets 1; planner/researcher/critic get 0. Each retry gets the failure log + previous attempt's output as additional context. After retries exhausted, surface to user with full swarm state.
- **Captured:** spec §"Failure handling / retry".

### Model routing: both subagent pinning + per-tool cascade
- **Why:** Subagent pinning (cheap for read-only, main for builder) gives the biggest cost win. Per-tool cascade (Haiku for read, Sonnet for edit) adds another layer. Both together is the v0.5.0 spec's intent, but the v0.5.0 release only applied it at the call level (not subagent level). v0.6.0 applies both.
- **Decision:**
  - Subagent model pinning: planner/researcher/critic/test-runner → cheap (Haiku-class); builder → main (Sonnet-class).
  - Per-tool cascade inside each subagent (v0.5.0 cascade map applied universally).
  - Model selection lives in `packages/swarm/src/optimizer-integration.ts`.
- **Captured:** spec §"Model routing (cascading)".

### Orchestrator fully autonomous; `--plan` on demand
- **Why:** Sid prefers low-friction (matches v0.5.0 style and pi-mono "YOLO" philosophy). Long-horizon tasks can opt into plan review via `--plan`.
- **Decision:** `pi swarm "<goal>"` runs to completion. `pi swarm --plan "<goal>"` shows plan + roster first, then runs. `pi swarm --dry-run "<goal>"` shows plan only.
- **Captured:** spec §"CLI surface".

### Multica preserved as one-shot direct dispatch
- **Why:** The 8 Multica named agents (jorvis, jouono, scout, summit, quill, surge, cipher, forge) are a fast path for single-purpose work. Routing them through the orchestrator would add cost + latency for no benefit.
- **Decision:** `pi multica <name> "<task>"` dispatches directly to a single subagent with model pinning. Bypasses orchestrator + scratchpad + worktrees.
- **Captured:** spec §"CLI surface".

### Verification gate = test-runner exit code
- **Why:** Deterministic verification is cheap and reliable. LLM-judge verification (e.g. "did this fix the bug?") is more expensive and not always right. Human-in-the-loop is the strongest but slowest.
- **Decision:** `verification-gate.ts` parses test-runner exit code + output. Test passes → orchestrator accepts builder output and proceeds to critique. Test fails → orchestrator retries builder with failure log.
- **Captured:** spec §"Components §1.5".
