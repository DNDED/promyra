## 5. Communication (file scratchpad)

Per the spec, file-based beats message bus for coding tasks: durable, diffable, debuggable, survives restarts.

Layout under `.pi-pro/swarm/<swarm_id>/`:

```
plan.md              # written by orchestrator; consumed by all subagents
planner.md           # written by planner; consumed by builder + critic
researcher.md        # written by researcher; consumed by planner + builder
builder.md           # written by builder; consumed by critic + test-runner
critic.md            # written by critic; consumed by orchestrator
test-results.txt     # written by test-runner; consumed by orchestrator (verification gate)
cost.json            # written incrementally by all subagents; consumed by budget.ts
state.json           # orchestrator state machine (idle/plan/dispatch/verify/done)
```

Each subagent reads its inputs from the scratchpad before running; writes its output to the scratchpad after. The orchestrator reads the outputs and dispatches the next phase.

## 6. Swarm topology (orchestrator sequence)

```
1. PLAN     — orchestrator writes plan.md; lists subagent roster + topo order
2. RESEARCH — researcher (cheap) reads repo, writes researcher.md
3. PLAN2    — planner (cheap) reads researcher.md, writes planner.md
             (steps 2 + 3 can run in parallel)
4. BUILD    — builder (main) reads plan + researcher + planner, writes builder.md,
             edits files in worktree
5. TEST     — test-runner (cheap) reads builder.md, runs tests in builder's worktree,
             writes test-results.txt
6. CRITIQUE — critic (cheap) reads builder.md + test-results.txt, writes critic.md
7. DECIDE   — orchestrator: test passed? → MERGE. Failed? → retry BUILD (max 2).
             Critic found issues? → retry BUILD.
8. MERGE    — orchestrator merges builder's worktree back to main
```

Parallelism: phases 2 and 3 can run in parallel (researcher and planner are both read-only, different inputs). Phases 4-6 are sequential (build → test → critique). The whole swarm is 3 LLM calls wide at peak.

## 7. Worktree layout

```
.pi-pro/
  swarm/
    <swarm_id>/
      plan.md, *.md, cost.json, state.json
  worktrees/
    <swarm_id>/
      builder/      # git worktree of cwd at branch swarm/<id>/builder
      test-runner/  # git worktree; runs tests against builder's branch (read access)
```

Planner, researcher, critic: no worktree (read-only on cwd). Builder writes to `worktrees/<id>/builder/`. Test-runner reads from there. At end, `pi swarm --merge <id>` merges builder's branch back to main.

## 8. Cost cap (per swarm)

- Default budget: $2.00; configurable `--budget=<usd>`.
- Soft warn at 50% — log warning + show in TUI swarm panel.
- Hard kill at 100% — current LLM call finishes (no mid-call abort), then swarm halts. State saved as `paused: budget exceeded`. Resume with `--budget=<higher>`.
- Budget accumulator: reads `cost.json` from scratchpad on every subagent completion.
- Per-subagent implicit caps (cheap models = hard ceiling on cost-per-call).

## 9. Failure handling / retry

- Per-subagent retry policy: `builder` gets 2 retries; `test-runner` gets 1; `planner` / `researcher` / `critic` get 0.
- Each retry gets the failure log + the previous attempt's output as additional context.
- `MAX_RETRIES` configurable per swarm via `--max-retries=N`.
- If retries exhausted, orchestrator surfaces to user with the full swarm state + the failing subagent's evidence.
- A `pi swarm --continue <id>` command reads the saved state, lets the user fix or rerun individual subagents.

## 10. Model routing (cascading)

- Per-subagent model pinning (v0.4.0 modelMap pattern, extended):
  - `planner`, `researcher`, `critic`, `test-runner` → cheap (Haiku-class)
  - `builder` → main (Sonnet-class)
- Per-tool cascade inside each subagent (v0.5.0 cascade map, applied universally):
  - `read`/`grep`/`glob` → cheap
  - `edit`/`write`/`apply_patch` → main
  - `bash` (test runners) → main
  - `bash` (install/lint) → cheap
- Model selection logic lives in `packages/swarm/src/optimizer-integration.ts` (wraps `packages/optimizer`'s cascade router).

## 13. Error handling

| Failure | Behavior |
|---|---|
| Orchestrator's plan.md invalid (LLM emitted garbage) | Orchestrator retries its own plan emit (1 retry) |
| Subagent fails after retries | Orchestrator surfaces to user; swarm halts in `paused: subagent failed` state |
| Test-runner exit code != 0 | Verification gate fails; orchestrator retries builder with failure log |
| Critic finds issues | Orchestrator retries builder with critique (only if test passed; otherwise test failure takes priority) |
| Budget exceeded | Hard kill; swarm halts in `paused: budget exceeded` state; resume with higher budget |
| Worktree creation fails (dirty tree, etc.) | Orchestrator surfaces immediately; no resources leaked (try/finally) |
| Subagent throws (network, parse error) | Retry (per-subagent policy); eventually escalate |
| Optimizer throw on a subagent | Subagent falls back to raw LLM call (v0.5.0 behavior) |
| Scratchpad write fails (disk full, perms) | Hard error; swarm halts; no state corruption (atomic write) |

## 15. Risks & Mitigations

1. **Orchestrator adds LLM overhead** (one extra Sonnet call per swarm). Mitigated by the orchestrator being 1 call (the plan), not N. *Net cost increase should be <30%.*
2. **Worktree creation is slow on large repos.** Mitigated by reusing the worktree across retries in the same swarm. Measure on bench.
3. **Critic adds cost without always adding value.** Mitigated by making critic optional via `--no-critic` flag; default on for complex tasks.
4. **Multica named agents lose their single-purpose fast path** if users run them through the orchestrator. Mitigated by keeping `pi multica <name>` as a direct subagent dispatch (bypasses orchestrator).
5. **File scratchpad gets large for long swarms.** Mitigated by subagents writing only their own output file + a rolling summary in `state.json`.
6. **Subagent nesting.** Spec'd as recursion depth = 1 (subagents do not spawn subagents). Tested.
